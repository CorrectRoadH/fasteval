`fasteval.config.ts` is the single file that controls how fasteval discovers, runs, and reports your evals. You export a configuration object from it using `defineConfig`, which provides type-safety and autocompletion for every option. Place the file at the root of your repository alongside `package.json`.

## Basic setup

```ts
// fasteval.config.ts
import { defineConfig } from "fasteval";
import { Console, JUnit } from "fasteval/reporters";

export default defineConfig({
  judge: { model: "anthropic/claude-haiku-4-5" },
  reporters: [Console(), JUnit(".fasteval/junit.xml")],
  maxConcurrency: 8,
  timeoutMs: 300_000,
  sandbox: "auto",
});
```

## Top-level options

### Agent registration

<ParamField body="agents" type="Agent[]" default="[]">
  An array of agent adapters to register with fasteval. Each entry is an object
  created with `defineAgent` or `defineSandboxAgent`. Built-in coding agents
  (`claude-code`, `codex`) do not need to be listed here — they are available
  by name automatically.

  ```ts
  import myAgent from "./agents/my-agent.js";
  import supportBot from "./agents/support-bot.js";

  export default defineConfig({
    agents: [myAgent, supportBot],
  });
  ```
</ParamField>

<ParamField body="defaultAgent" type="string">
  The name of the agent to use when `--agent` is not passed on the CLI. Must
  match the `name` field of a registered agent, or be a built-in agent name
  such as `"claude-code"` or `"codex"`.

  ```ts
  export default defineConfig({
    defaultAgent: "claude-code",
  });
  ```
</ParamField>

---

### Judge

<ParamField body="judge.model" type="string">
  The model to use for LLM-as-judge scoring (the `t.judge.*` assertions). This
  should be a different model from the agent under test so that the judge
  remains independent. Accepts any model identifier your judge provider
  understands.

  ```ts
  export default defineConfig({
    judge: { model: "gpt-5.4-mini" },
  });
  ```
</ParamField>

---

### Reporters

<ParamField body="reporters" type="Reporter[]" default="[Console()]">
  An array of reporter instances. Reporters consume eval results as they
  complete and produce output (console summaries, XML files, etc.). fasteval
  ships two built-in reporters:

  - **`Console()`** — streams results to stdout in real time as evals complete.
  - **`JUnit(path)`** — writes a JUnit XML report to `path` when the run
    finishes. Compatible with GitHub Actions, CircleCI, and other CI systems.

  ```ts
  import { Console, JUnit } from "fasteval/reporters";

  export default defineConfig({
    reporters: [Console(), JUnit(".fasteval/junit.xml")],
  });
  ```
</ParamField>

---

### Concurrency & performance

<ParamField body="maxConcurrency" type="number" default="varies">
  Maximum number of eval attempts to run in parallel across the entire run.
  The runner maintains a pool of exactly this many in-flight attempts, filling
  slots as earlier attempts complete. Can be overridden at runtime with
  `--max-concurrency`.

  ```ts
  export default defineConfig({
    maxConcurrency: 8,
  });
  ```
</ParamField>

<ParamField body="sandboxConcurrency" type="number">
  Maximum number of sandbox instances (Docker containers or cloud VMs) to run
  in parallel. Use this to avoid overwhelming the sandbox backend when
  `maxConcurrency` is high. Particularly relevant for local Docker setups where
  each container consumes memory and CPU.

  ```ts
  export default defineConfig({
    maxConcurrency: 6,
    sandboxConcurrency: 2,
  });
  ```
</ParamField>

---

### Timeouts & budget

<ParamField body="timeoutMs" type="number" default="300000">
  Per-eval timeout in milliseconds. If an eval takes longer than this value,
  the runner force-cancels it and marks it `failed` with `error: timeout`.
  Can be overridden at runtime with `--timeout`.

  ```ts
  export default defineConfig({
    // 10 minutes per eval
    timeoutMs: 600_000,
  });
  ```
</ParamField>

<ParamField body="budget" type="number">
  Cost limit in USD for a run. The runner accumulates estimated spend across all
  dispatched attempts. Once the total exceeds `budget`, no new attempts are
  dispatched; any in-flight attempts finish normally. Can be overridden at
  runtime with `--budget`.

  ```ts
  export default defineConfig({
    budget: 5.00,
  });
  ```
</ParamField>

---

### Sandbox

<ParamField body="sandbox" type='"docker" | "vercel" | "auto"' default='"auto"'>
  The sandbox backend to use for coding-agent evals that need filesystem
  isolation.

  | Value | Behavior |
  |---|---|
  | `"docker"` | Spin up a local Docker container. Requires Docker to be installed and running. No cloud credentials needed. |
  | `"vercel"` | Use Vercel Sandbox (a cloud-hosted micro-VM). Requires `VERCEL_TOKEN` or `VERCEL_OIDC_TOKEN` to be set in the environment. |
  | `"auto"` | Detect automatically at runtime: use `vercel` if `VERCEL_TOKEN` or `VERCEL_OIDC_TOKEN` is present, otherwise fall back to `docker`. |

  Can be overridden at runtime with `--sandbox`.

  ```ts
  export default defineConfig({
    sandbox: "docker",
  });
  ```
</ParamField>

---

### Workspace

<ParamField body="workspace" type="string">
  Path to the workspace directory that gets uploaded into the sandbox for
  every coding-agent eval. The runner copies this directory's contents into
  each sandbox before running the agent. Relative paths are resolved from the
  repository root.

  ```ts
  export default defineConfig({
    workspace: "./workspaces/next-app",
  });
  ```
</ParamField>

### Lifecycle hooks

Hooks let you run setup and teardown logic at two scopes: once for the entire run, and once per eval attempt (around the sandbox). All teardown hooks run inside `finally` blocks, so they execute even when an eval fails.

#### Run-scoped hooks

<ParamField body="hooks.run.setup" type="(ctx) => Promise<void> | void">
  Called once before the first attempt of the run. Use this to start shared
  infrastructure that all evals depend on — a mock API server, a shared
  database, a connection pool. If this hook throws, the entire run is aborted.

  ```ts
  export default defineConfig({
    hooks: {
      run: {
        async setup(ctx) {
          await mockApiServer.start();
        },
        async teardown(ctx) {
          await mockApiServer.stop();
        },
      },
    },
  });
  ```
</ParamField>

<ParamField body="hooks.run.teardown" type="(ctx) => Promise<void> | void">
  Called once after the last attempt of the run completes (always, even on
  failure). Use this to shut down shared resources started in `hooks.run.setup`.
  Errors here are recorded as diagnostics and do not change any eval verdict.
</ParamField>

#### Sandbox-scoped hooks

<ParamField body="hooks.sandbox.setup" type="(sandbox, ctx) => Promise<(() => Promise<void>) | void> | void">
  Called before each individual eval attempt, after the sandbox has been
  created and the workspace uploaded. Use this to write `.env` files, install
  additional dependencies, or start per-attempt services. You can read
  `ctx.flags` to branch on experiment feature flags.

  The function may return a cleanup closure; if it does, that closure is called
  during teardown (equivalent to defining `hooks.sandbox.teardown`).

  If this hook throws, the error is isolated to the current eval (marked
  `failed`) and does not affect other running evals.

  ```ts
  export default defineConfig({
    hooks: {
      sandbox: {
        async setup(sandbox, ctx) {
          await sandbox.writeFiles({
            ".env": `API_KEY=${process.env.API_KEY}`,
          });
          // Optionally return a cleanup closure
          return async () => {
            // cleanup runs after the eval, even on failure
          };
        },
        async teardown(sandbox, ctx) {
          // Alternative to returning a closure from setup
        },
      },
    },
  });
  ```
</ParamField>

<ParamField body="hooks.sandbox.teardown" type="(sandbox, ctx) => Promise<void> | void">
  Called after each eval attempt finishes, always inside a `finally` block.
  Use this to clean up resources created in `hooks.sandbox.setup`. Errors here
  are recorded as diagnostics only and do not change the eval verdict.
</ParamField>

<Note>
  Config hooks run before experiment hooks when both are defined. Teardown
  hooks always run in reverse order (experiment teardown first, then config
  teardown).
</Note>

## Built-in reporters

<Accordion title="Console()">
  Streams results to stdout as evals complete, with real-time progress. Shows
  each eval's ID, verdict (pass / fail / scored / skipped), duration, and
  failure details. Prints a summary line at the end of the run.

  ```
  Discovered 3 evals

    ✓ classify (12ms)
    ✓ weather/brooklyn (456ms)
    ✗ fixtures/button (38s)
      - gate: EVAL.ts › accepts label / onClick [FAILED]
        Expected src to contain "onClick"

  Results:  2 passed, 1 failed, 0 scored, 0 skipped
  ```

  Import and instantiate with no arguments:

  ```ts
  import { Console } from "fasteval/reporters";

  reporters: [Console()]
  ```
</Accordion>

<Accordion title="JUnit(path)">
  Writes a JUnit XML report to `path` when the run finishes. Each eval becomes
  a `<testcase>` element; failures and errors include the message text.
  Compatible with GitHub Actions test reporting, CircleCI, Jenkins, and any
  other CI system that ingests JUnit XML.

  ```ts
  import { JUnit } from "fasteval/reporters";

  reporters: [JUnit(".fasteval/junit.xml")]
  ```

  You can combine both reporters to get live console output and a CI-friendly
  artifact in the same run:

  ```ts
  reporters: [Console(), JUnit(".fasteval/junit.xml")]
  ```
</Accordion>

## Environment variables

fasteval reads the following environment variables at runtime. You do not configure these in `fasteval.config.ts` — set them in your shell or CI secrets.

| Variable | Purpose |
|---|---|
| `VERCEL_TOKEN` | Authenticates with Vercel Sandbox when `sandbox` is `"vercel"` or `"auto"`. |
| `VERCEL_OIDC_TOKEN` | Alternative OIDC-based token for Vercel Sandbox (used in CI environments that support OIDC). |
| `ANTHROPIC_API_KEY` | Required by the built-in `claude-code` agent adapter. |
| `OPENAI_API_KEY` | Required by the built-in `codex` agent adapter. |

<Warning>
  Do not commit API keys to your repository. Use your CI system's secret
  management (e.g. GitHub Actions Secrets) and read them as environment
  variables in your workflow.
</Warning>

## Complete annotated example

The following config shows all commonly used options together with explanatory comments:

```ts
// fasteval.config.ts
import { defineConfig } from "fasteval";
import { Console, JUnit } from "fasteval/reporters";
import myAgent from "./agents/my-agent.js";

export default defineConfig({
  // Register your own agent adapters.
  // Built-in agents (claude-code, codex) don't need to be listed.
  agents: [myAgent],

  // Use this agent when --agent is not passed on the CLI.
  defaultAgent: "claude-code",

  // LLM-as-judge model — keep this separate from the agent under test.
  judge: { model: "gpt-5.4-mini" },

  // Print results to the console and write a JUnit XML artifact for CI.
  reporters: [Console(), JUnit(".fasteval/junit.xml")],

  // Run up to 8 eval attempts in parallel.
  maxConcurrency: 8,

  // Limit sandbox instances to avoid overwhelming Docker locally.
  sandboxConcurrency: 3,

  // Cancel any eval that takes longer than 5 minutes.
  timeoutMs: 300_000,

  // Use Docker locally; switch to Vercel Sandbox when VERCEL_TOKEN is set.
  sandbox: "auto",

  // Upload this directory into every sandbox before running the agent.
  workspace: "./workspaces/next-app",

  // Stop dispatching new attempts once cumulative spend exceeds $10.
  budget: 10.00,

  hooks: {
    run: {
      // Start shared infrastructure once before any evals run.
      async setup(ctx) {
        await mockServer.start({ port: 4000 });
      },
      // Shut it down after all evals finish (runs even on failure).
      async teardown(ctx) {
        await mockServer.stop();
      },
    },
    sandbox: {
      // Write per-attempt config into each sandbox before the agent runs.
      async setup(sandbox, ctx) {
        await sandbox.writeFiles({
          ".env": `MOCK_SERVER_URL=http://host.docker.internal:4000`,
        });
      },
    },
  },
});
```

## Real-world example: coding-agent-memory-evals

This is the actual `fasteval.config.ts` from the [coding-agent-memory-evals](https://github.com/CorrectRoadH/coding-agent-memory-evals) project. It evaluates Claude Code against a Next.js workspace using a local Docker sandbox, with a GPT judge for LLM-as-judge assertions:

```ts
import { defineConfig } from "fasteval";

export default defineConfig({
  defaultAgent: "claude-code",
  sandbox: "docker",
  workspace: "./workspaces/next-app",
  judge: { model: "gpt-5.4-mini" },
  timeoutMs: 600_000,
  maxConcurrency: 3,
  sandboxConcurrency: 2,
});
```

Key decisions in this config:

- **`defaultAgent: "claude-code"`** — no need to pass `--agent` on every invocation; `claude-code` is a built-in adapter so it does not appear in `agents: [...]`.
- **`sandbox: "docker"`** — all evals run in local Docker containers; no cloud credentials required.
- **`workspace: "./workspaces/next-app"`** — the Next.js starter is uploaded into every sandbox before the agent runs.
- **`judge: { model: "gpt-5.4-mini" }`** — a cheap, fast model is used for judge scoring, independent of the agent under test.
- **`timeoutMs: 600_000`** — coding tasks on a real Next.js project can take up to 10 minutes.
- **`maxConcurrency: 3` / `sandboxConcurrency: 2`** — limits keep local Docker resource usage manageable while still running multiple evals in parallel.
